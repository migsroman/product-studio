How Product Studio auto-generates the UW workflow from rater schema + product configuration

1
Field Inventory
Steps 1–3 → Automatic

The system builds a classified inventory of every field the UW workflow needs to collect. Each field is tagged with metadata from the rater schema and product configuration.

📄 From Rater Schema (Step 1–2)

Each mapped field inherits: sheet origin (which rater sheet it came from), data type (string, number, boolean, date, currency), cell reference, and description.

3
Auto-Generation Engine
System logic — the core intelligence

The system populates each workflow step with fields, infers field configurations, generates conditional rules, and creates form triggers — all automatically. The PM lands in Step 4 with a pre-built workflow that's ~80% ready.

⚙️ Field Config Inference

For each field, the engine infers:

  • Input type from rater data type (boolean → toggle, date → date picker, string with known values → dropdown)
  • Allowed values from Step 3 product config (coverage limit options, deductible options, class codes)
  • Default value from Step 3 defaults
  • Label from rater field description
  • Required flag from coverage required status
🔀 Conditional Rule Inference

The engine auto-creates conditions from:

  • Coverage dependencies: if field is tagged coverage:med_pay and a med_pay_selected toggle exists → show when toggle = true
  • Unit-level dependencies: if gvw_gcw only applies to truck classes → show when vehicle_class in [truck types]
  • Coverage-gated deductibles: deductible_comp → show when comprehensive coverage enabled in Step 3
📎 Form Trigger Inference

The engine auto-creates form triggers from:

  • Coverage → endorsement mapping: med_pay_selected = true → attach CA 99 17
  • Product option → form mapping: hired_selected = true → attach CA 20 48
  • These rules come from a form library that maps endorsements to their triggering conditions
📐 Step Validation

The engine checks completeness:

  • Every active rater field is assigned to a step
  • No field appears in multiple steps
  • Required fields have input configs
  • Coverage-dependent fields have conditions

4
PM Review & Customization
Human-in-the-loop — Step 4 in the UI

The PM reviews the auto-generated workflow and makes targeted adjustments. The system did the heavy lifting; the PM validates, refines, and adds business rules the system couldn't infer.

✅ Review auto-generated configs

Verify field labels, input types, and allowed values are correct. Override where needed (e.g., change a text input to a dropdown with specific values).

🔄 Reorder & reorganize steps

Move steps up/down. Move fields between steps (e.g., pull a field from Business Info into Coverage Selection). Add custom steps.

🔀 Add custom conditions

Add business logic the system couldn't infer. E.g., "Show towing coverage option only when vehicle_class is not Trailer."

📎 Add custom form triggers

Link endorsements to workflow selections that aren't captured in standard coverage-to-form mappings. E.g., a state-specific filing requirement.

The key principle

The Workflow Builder is not a blank canvas. It's a review-and-refine experience. The system does 80% of the work; the PM adds the 20% that requires product expertise and business judgment.

End-to-End Summary

The complete flow from rater upload to published UW workflow.

1

Rater Connection

Upload workbook or connect API. System extracts input/output schema.

Produces: raw schema
2

Schema Mapping

Map rater fields to product data model. Classify each field by origin and type.

Produces: classified fields
3

Product Config

Define what the product offers. Narrows the field inventory and provides allowed values.

Produces: active inventory + options
⚡

Auto-Generation

System populates workflow: assigns fields, infers configs, creates conditions & form triggers.

Produces: pre-built workflow
4

Review & Customize

PM reviews, reorders, overrides, and adds custom logic. Then previews the live UW experience.

Produces: published workflow

The workflow builder is not a blank canvas — it's a review-and-refine experience.

The system does the heavy lifting by inferring field configs, conditions, and form triggers from the rater schema and product configuration. The PM adds the 20% that requires product expertise.
